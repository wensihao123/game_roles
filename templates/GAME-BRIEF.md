# GAME-BRIEF — <游戏名/代号>

> **这是什么**:立项仪式(`/g-kickoff`)的产物,整个游戏的"承诺书"。与 project-context.md
> 并列为**人人必读**的常驻文件:project-context 回答"这个项目怎么施工",GAME-BRIEF 回答
> "我们到底在做什么游戏、凭什么、做到哪为止"。
>
> **状态纪律**:立项后它是"当前承诺",只写当前真相。支柱/范围要变更时走 producer 决策,
> 就地更新本文件(不追加历史);重大转向在 BACKLOG Decision log 留一行痕。
>
> **怎么用**:所有 role 冷启动必读 §1/§4/§5(一句话、支柱、范围);designer 每份
> FEATURE-DESIGN 必须对照 §4 回答"支持哪些支柱/牺牲哪些";producer 用 §5 守范围;
> playtest 用 §2/§6 出验证假设。

## 1. 一句话

- 这是个什么游戏,给谁玩:<类型 + 核心动词 + 目标人群,一句话说完。
  例:高难度但宽容的 2D 平台跳跃,给硬核手感党>

## 2. 核心体验

- **玩家幻想(Fantasy)**:玩家想成为谁 / 做到什么?<例:我是靠自己手上功夫翻盘的人>
- **情绪目标(Emotional Goal)**:主导情绪是什么?<紧张后的释放 / 掌控感 / 治愈 / 探索的悸动…>
- **核心循环(Core Loop)**:动作 → 结果 → 反馈 → 再来一次的动机
  <例:看到缺口 → 起跳 → 险过或差一点 → 立刻重试 → 熟练度上升>

## 3. 目标玩家

- 给谁玩:<画像,不要写"所有人">
- 他们现在在玩什么:<同类参照,证明这群人存在>
- 凭什么值得他们的时间:<你的游戏对他们的独特承诺;答不出来就该回炉>

## 4. 设计支柱(2~4 条,每条必须带取舍面)

> 支柱是评审的尺子,不是口号。**每条都要写清"为了它我们接受牺牲什么"**——
> 写不出牺牲面的支柱是废话,删掉重想。
